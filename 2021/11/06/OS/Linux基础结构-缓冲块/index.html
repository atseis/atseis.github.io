Linux基础结构——缓冲块
2021-11-06|课程笔记操作系统|
Word count: 1.4k|Reading time: 5 min

前言

  • Linux 0.11

在由进程0切换到进程1后，首先要做的就是为安装硬盘文件系统做准备，而其中就涉及到许多与缓冲块有关的操作：

  • 读取硬盘的引导块到缓冲区
  • 将找到的缓冲块与请求项挂接

结构

基础结构

缓冲区位于 内核代码块 和 主内存区 间

  • 从内核模块末端 end 开始： end 值是链接期间设置的值

整个缓冲区：

  • 高端被划分成一个个 1024B 的缓冲块
  • 低端分别建立起对应各缓冲块的头结构 buffer_head

链表结构

其中对于理解结构比较重要的就是最后5个指针：

  • b_wait：等待缓冲块的任务队列，这里是一个隐式队列
  • b_prev, b_next：与 hash 值有关的指针，在下图中对应与横向的箭头
  • b_prev_free, b_next_free：在 free_list 队列中的指针，对应于下图中虚线

可以看出，有两条主线：

  • 散列值对应链表：便于查找当前
  • 当前空闲缓冲块链表

insert, remove

进、出队列都涉及到对于上述2条线的操作：

  • remove_from_queues
  • insert_into_queues

且都是放在“最前面”：

  • 对于环形的 freelist 而言，放在 freelist 的前一个，但是不改变 freelist 指针本身。这也就相当于放在了“最后”
  • 对于 hash 的情况，则是直接放在队首

hash 表

  • 将 (dev, block) 看做是将要匹配的硬件部分的 id
  • 而这样的 id 有很多，不可能每个都维护一个对应的缓冲块，因此定义一种 hash 算法，对于每一种可能的 hash 值维护一个缓冲块，这样就构成了一个 hash 表： hash_table[NR_HASH]
    • 这样就不可避免的产生冲突：即不同的 “id” 会产生同样的 hash 值
  • 这样构成的数组表，本身就有了一层“链接”：即上图中横向的箭头
  • 但为了便于查询空闲缓冲块，还需要在数组的基础上引入另一条“线”，因此再维护一个空闲块间的链表，即上图中虚线部分

总结上述2个函数：

  • get_hash_table(dev, block) ： 从 hash 表中获取对应buffer head, 最后获取到的（如果有的话）一定是与 (dev, block) 匹配的，且可以立即使用的
  • find_buffer(dev, block)： 只是简单地在 hash 表中查询，最后获取到的（如果有的话）只能保证与 (dev, block) 匹配，但不能保证能立即使用
