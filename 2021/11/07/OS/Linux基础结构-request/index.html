Linux基础结构——request
2021-11-07|课程笔记操作系统|
Word count: 1.6k|Reading time: 6 min

前言

Linux 0.11 中有两种外设：

  • 块设备
  • 字符设备

进程与块设备沟通必须===>必须经过 mem 中的缓冲区

  • 缓冲区中缓冲块<---->块设备上的逻辑块 ===> 管理r/w的数据结构就是请求项 request[32]

数据结构

设备

首先是内核对设备的管理

  • 通过一张设备表管理，每 种 设备在表中占有一项
  • 每一项有2个属性
    • 请求项操作的函数指针
    • 当前请求项指针

这很好理解，即对于每一种设备：

  • 内核应该怎样和它打交道，如对于硬盘驱动程序，就是依靠 do_hd_request() 和它打交道
  • 当前需要它做什么操作

数据结构如下：

序号 设备
0 unused (nodev)
1 /dev/mem
2 /dev/fd
3 /dev/hd
4 /dev/ttyx
5 /dev/tty
6 /dev/lp
7 unnamed pipes

请求项request

其次是 request. 其相当于是对“动词”，或“操作”的一种封装

从后几个属性可以看到， request 的操作涉及广泛：

  • 缓冲块，缓冲头
  • 任务
  • 自身的链表连接

请求项队列

顾名思义，就是由请求项组成的队列，不过与 buffer_head 类似，虽然是采用数组形式进行组织，但其间的逻辑仍有对应的指针（即 struct request * next）进行连接

这里首先需要厘清2个函数：

  • bread：上层程序访问块设备数据通过 bread() 向缓冲区管理程序申请
  • lw_rw_block： 如果所需的数据暂时不在缓冲区，则管理程序会通过 ll_rw_block 向块设备驱动程序申请，同时让程序对应的进程睡眠等待

结构如下所示：

也就是说， 直接与设备打交道的是 ll_rw_block()!

当内核发出一个块设备读写或其它操作请求时：

  • ll_rw_block() 函数即会
    • 根据其参数中指明的操作命令和数据缓冲块头中的设备号
    • 利用对应的请求项操作函数 do_XX_request() 建立一个块设备请求项
    • 并利用电梯算法插入到请求项队列中

请求项队列由请求项数组中的项构成，共有 32 项

请求命令类型

即 request 结构中 cmd 属性

主要有4种：

值得注意的是 READA, WRITEA 2种，

  • 2种都是表示“预”操作，即预读、预写
  • 在 make_request 中
    • 当指定缓冲区被上锁时，放弃预读、写请求
    • 否则视作普通的 READ/WRITE

  • 将操作封装成数据结构，便于管理大量操作同时发出的情形
