Linux基础逻辑——读硬盘
2021-11-08|课程笔记操作系统|
Word count: 1.8k|Reading time: 7 min

前言

本文主要以读盘操作为例分析。

首先回顾一下过程：(以下部分中，==>表示先后次序，–>表示调用关系)

  • 由进程0轮转到进程1执行后，fork()==>init()
  • init() --> setup((void*)&drive_info)
  • setup() === sys_setup(void* BIOS) setup 是系统调用，通过 int 0x80
  • sys_setup(void* BIOS) --> bread(0x300+drive*5, 0) 需要读取引导块信息
    • 1 个物理硬盘最多可分为 4 个逻辑盘， 0:物理盘， 1~4: 逻辑盘
    • 第 1 个物理盘号是 0x300，第 2 个是0x305
  • bread(dev, block) --> getblk(dev, block) 想要得到一个与 dev, block 相符或者空闲的缓冲块，如果 bh->b_uptodate 为真则直接返回。因为是第一次，必然是不满足的，因此会继续向下执行内容
  • getblk(dev, block) ==> ll_rw_block(READ, bh) 将找到的缓冲块与请求项挂接
    • ll_rw_block 先判断缓冲块对应的设备是否存在 或 这个设备的请求项函数是否挂接正常
    • 满足条件才继续向下执行， 否则报错，直接返回
    • 这一步很好理解：缓冲块相当于资源，但不能光有资源，还得知道有了资源之后做啥，这就得需要挂接请求项
  • ll_rw_block(rw, bh) --> make_request(major, rw, bh)
  • make_request(major, rw, bh) --> add_request(major+blk_dev, req) 在得到合适的 request 后，进行赋值，最后加入 request 队列
  • add_request(dev, req) --> (dev->request_fn)() 如果设备dev->current_request为空，证明目前该设备没有请求项，因此直接将设备的请求指针指向本 req, 并调用设备对应的请求函数，对于硬件来说，就是 do_hd_request()

do_hd_request 发送操作指令

  1. 分析当前request数据成员，解析出需要操作的磁头、扇区、柱面、操作多少个扇区。。。
  2. 建立硬盘读盘必要的参数，将磁头移到0柱面
  3. 给硬盘发送操作命令

重点关注第3步

可以看到， do_hd_request 通过 hd_out 去下达最后的硬盘操作指令

注意传入的最后2个参数：

  • WIN_READ：接下来要进行的是读操作
  • &read_intr： read_intr() 是读盘操作对应的中断服务程序
    • 因此提取函数地址，准备挂接

总结

硬盘产生中断后，中断服务程序执行怎样的中断处理程序在形如 hd_out 这样的函数中指定，如果是读盘则为 read_intr()

在指定的处理程序中也会对整个流程的结束与否进行判断，如果没有结束，则会再次挂接：

  • 也就是说，一次挂接只处理一次
  • 如果一次挂接不能解决问题，就会连续依次挂接，如同老师课上举的比喻“打机枪”一样

而在结束请求时则会唤醒相关等待的进程，这与 sleep_on 构建的隐式队列又产生呼应。
